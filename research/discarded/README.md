# Discarded Research

`discarded/`는 현재 연구 방향에서 사용하지 않기로 판정한 설명, 구분, 연구 흐름, 프롬프트 설계 아이디어를 보존한다.

폐기는 삭제와 다르다.

- 현재 연구의 근거로 사용하지 않는다.
- 실패 이유를 남긴다.
- 전체가 틀렸더라도 남아 있는 문제의식이나 반례를 보존한다.
- 다시 사용할 때에는 어디서 왜 회수했는지 밝힌다.
- 이름만 바꾸어 조용히 mainline으로 되돌리지 않는다.

---

## 1. 폐기할 수 있는 대상

- 하나의 이론 설명
- 개념 구분
- 연구 thread 전체
- 여러 문제를 묶은 방식
- 프롬프트 구현 전략
- 평가 해석
- 과도한 통합
- 불필요한 형식화

---

## 2. 폐기 이유를 구분한다

### 개념 중복

다른 설명이 같은 현상을 더 적은 가정으로 설명한다.

### 잘못된 구분

실제로 함께 움직이는 것을 인위적으로 분리했거나, 서로 다른 것을 하나로 묶었다.

### 반례

핵심 주장을 직접 약화하는 관찰이나 자료가 있다.

### 예측 실패

이론이 기대한 수행 차이가 나타나지 않는다.

### 구현 의존

인간 구조라고 생각했지만 특정 프롬프트 엔진이나 표현 방식의 제약이었다.

### 비판별성

어떤 결과가 나와도 설명할 수 있어 경쟁 이론과 구분되지 않는다.

### 과도한 비용

설명력에 비해 개념과 구조가 지나치게 복잡하다.

### 질문 소실

처음 설명하려던 현상보다 체계 자체를 유지하는 일이 중심이 되었다.

---

## 3. 권장 기록 형식

```text
무엇을 설명하려 했는가
당시 왜 유망하다고 생각했는가
어떤 자료와 모델에 연결되었는가
어떤 문제 또는 반례가 드러났는가
이론 실패인가 구현 실패인가
현재 무엇이 이를 대체하는가
여전히 유효한 문제의식은 무엇인가
재검토할 조건이 있는가
관련 transition 기록은 무엇인가
```

---

## 4. dormant와의 차이

```text
dormant
= 아직 판정할 수 없거나 지금 밀지 않음

discarded
= 현재는 사용하지 않는 것이 더 타당하다고 판정함
```

불확실한 것을 성급히 폐기하지 않고, 실패한 것을 무기한 dormant로 숨기지도 않는다.

---

## 5. 폐기된 구조의 회수

폐기된 자료 일부를 다시 사용할 수 있다.

예:

```text
전체 모델은 실패했지만 특정 반례는 유효함
개념 이름은 틀렸지만 설명하려던 현상은 남음
프롬프트 구현은 실패했지만 경계 조건은 유용함
한 축으로는 실패했지만 다른 thread의 질문으로 회수 가능함
```

회수할 때에는 다음을 남긴다.

- 무엇을 회수했는가
- 무엇은 계속 폐기 상태인가
- 현재 맥락에서 의미가 어떻게 달라졌는가
- 과거 실패가 다시 발생하지 않도록 무엇을 확인할 것인가

이 디렉터리는 실패를 숨기는 쓰레기통이 아니라 같은 실패를 반복하지 않기 위한 연구 기억이다.
