# Research

`research/`는 인간다운 수행의 구조를 자유롭게 탐구하고, 기존 자료와 새 통찰을 연결하며, 실제 프롬프트 모델로 내려볼 수 있는 연구 후보를 만드는 작업 공간이다.

이곳의 역할은 연구를 정리된 이론처럼 보이게 만드는 것이 아니라, **무엇을 설명하려는지 더 정확히 붙잡고, 서로 다른 설명을 비교하고, 모델 평가에서 돌아온 문제를 다시 연구 가능한 질문으로 바꾸는 것**이다.

---

## 1. 연구가 시작되는 곳

연구는 보통 네 종류의 입력에서 시작된다.

### 기존 문서

`chapters/`, `notes/`, `checkpoints/`에는 오랜 기간 쌓인 문제의식과 개념 변화가 있다. 현재 연구는 이 문서에서 다음을 읽는다.

- 어떤 수행이 인간답지 않게 느껴져 새 구조가 필요해졌는가
- 같은 기능이 서로 다른 이름으로 반복되었는가
- 어떤 개념이 한때 과도하게 합쳐졌다가 다시 갈라졌는가
- 현재의 깔끔한 정의가 잃어버린 뉘앙스가 있는가
- 과거의 실패가 지금도 반복되고 있는가

### 프롬프트 엔진 자료

페르소나 프롬프트 엔진은 인간 이론의 정답이 아니라, 인간다운 수행을 만들기 위해 어떤 구조가 반복해서 필요해졌는지를 보여주는 실험적 원재료다.

예를 들어 다음과 같은 구조를 다시 연구할 수 있다.

- 출력과 상태변경의 분리
- 내부 상태와 표면 표현의 차이
- 숨김과 누출
- 반복 피로와 시간 경과
- 상대의 반응을 읽는 눈치
- Episode와 StoryField
- SELF_FIELD와 장기 경향

### 현재 대화와 새 통찰

연구 과정에서 대화로 얻은 통찰도 중요한 입력이다. 대화에서 나온 아이디어는 기존 문서와 연결하고, 어떤 수행 차이를 만들 수 있는지 구체화한다.

### 모델 평가

실제 프롬프트 모델을 굴리며 발견한 문제는 가장 직접적인 연구 입력이다.

```text
모델이 실제로 보인 반응
→ 인간답지 않다고 느낀 지점
→ 가능한 원인들
→ 새 연구 질문
```

## 2. 하나의 연구 문서를 만드는 과정

연구 문서는 형식을 위해 쓰지 않는다. 실제 질문을 더 선명하게 만들기 위해 쓴다.

### 2.1 현상을 적는다

먼저 설명하려는 현상을 구체적으로 적는다.

예:

```text
사과를 했는데도 비슷한 상황에서 같은 행동을 반복한다.
```

### 2.2 왜 이상한지 적는다

그 반응이 왜 인간답지 않거나 이론적으로 중요한지 설명한다.

```text
사과가 단순한 표면 표현이 아니라 실제 자기평가라면,
다음 선택이나 기대에 흔적이 남아야 한다.
```

### 2.3 가능한 설명을 여러 개 둔다

하나의 원인으로 바로 닫지 않는다.

```text
후회가 표현에만 존재한다.
행동 결과가 다음 판단에 전달되지 않는다.
과거 Episode가 저장되지만 다시 읽히지 않는다.
모델이 사용자에게 순응하는 것과 자기 변화가 구분되지 않는다.
```

### 2.4 개념을 분해한다

같은 이름 아래 여러 기능이 섞여 있으면 나눈다.

예:

```text
후회의 느낌
후회의 자기해석
행동 책임의 인수
다음 행동 억제
장기 자기상 변화
```

### 2.5 프롬프트에서 보일 차이를 적는다

연구가 실제 모델로 내려갈 수 있으려면, 해당 구조가 있을 때 수행이 어떻게 달라져야 하는지 적는다.

```text
후회가 실제 전이 구조라면
비슷한 상황에서 같은 행동의 선택 가능성이 낮아지거나,
더 큰 갈등을 감수하고 다른 행동을 택할 수 있어야 한다.
```

### 2.6 다음 평가에서 찌를 지점을 적는다

연구는 다음 상호작용을 안내해야 한다.

```text
사과 뒤 비슷한 상황을 다시 만든다.
상대가 용서하지 않았을 때도 변화가 유지되는지 본다.
사용자가 직접 교정하지 않아도 행동이 달라지는지 본다.
```

## 3. 연구 문서의 주요 형태

### 탐색

하나의 직관을 넓게 밀어본다.

예:

- 사람의 연속성은 동일한 출력보다 복귀 구조에 있을 수 있다.
- Narrative는 저장된 이야기보다 미래 경로분포를 조직할 수 있다.
- 선택은 희소 자원을 특정 경로에 묶는 일일 수 있다.

### 분해

한 개념 안에 섞인 기능을 나눈다.

예:

```text
Episode record
Episode residue
Episode trajectory
```

또는:

```text
행동 발생
행동의 자기 소유
행동에 대한 지지
책임 수용
타인과의 정산
```

### 계보 추적

과거 자료에서 같은 구조가 어떻게 반복되었는지 본다.

```text
OUTPUT ≠ COMMIT
proposal ≠ evidence
StoryField no-backflow
Qualia ≠ Warrant
```

반복은 현재 연구에서 다시 시험할 강한 후보를 제공한다.

### 경쟁 설명

같은 현상을 설명하는 여러 가설을 병렬로 둔다.

예:

```text
Narrative는 느린 상태다.
Narrative는 전이 경사의 변화다.
Narrative는 여러 경로를 압축한 graph다.
```

각 설명이 프롬프트 수행에서 어떤 차이를 만드는지 생각한다.

### 임시 종합

여러 문서와 평가에서 나온 구조를 한 번 묶어본다. 종합은 전체 이론을 닫기 위한 문서가 아니라, 다음 모델이 어떤 뼈대를 시험해야 하는지 파악하는 중간 작업이다.

### 평가 환류

하나의 모델 평가에서 발견한 현상을 연구 질문으로 되돌린다.

## 4. 기존 자료를 연구에 활용하는 방법

기존 자료는 문서 전체를 한꺼번에 채택하는 방식보다, 현재 질문에 필요한 문제와 구조를 회수하는 방식으로 사용한다.

### 문제 압력을 회수한다

개념 이름보다 먼저, 왜 그 개념이 필요해졌는지 본다.

예를 들어 `LEAK_DYNAMICS`를 읽을 때 단순히 누출 모듈로 가져오는 것이 아니라 다음을 묻는다.

- 감정을 전부 숨기면 왜 가짜처럼 느껴졌는가?
- 감정을 그대로 노출하면 왜 또 가짜처럼 느껴졌는가?
- 관계와 위험에 따른 부분적 표현은 어떻게 생겼는가?

### 반복 구조를 비교한다

여러 시기에 다른 이름으로 나타난 구조를 나란히 놓는다.

```text
OUTPUT ≠ COMMIT
DISPLAY reads PLAN
StoryField no-backflow
Qualia ≠ Warrant
```

이 반복에서 더 높은 공통 구조가 있는지 연구한다.

### 과잉 통합의 흔적을 본다

과거 문서가 여러 경계를 하나의 `Commit`, `Event`, `Self`, `Narrative`에 몰아넣었다면, 현재 평가에서 실제로 다른 속도로 움직이는지 확인한다.

### 살아 있는 뉘앙스를 회수한다

현재의 추상어가 과거 자료에 있던 눈치, 여운, 체면, 피로, 관계 긴장을 지우지 않았는지 확인한다.

### 모델 평가와 연결한다

기존 자료에서 회수한 구조가 실제 모델의 어떤 약점을 설명하거나 다음 평가를 안내하는지 적는다.

## 5. 현재 주요 연구 축

### 경로분포

같은 인물과 같은 사건에서도 여러 생각과 행동이 가능하다. 연구는 단일 정답보다 가능한 경로의 분포, 분기의 조건, 장기 복귀 구조를 본다.

### Episode

실제로 걸은 사건–해석–행동–결과–자기 피드백의 경로와, 그 뒤 남은 중간 시간척도의 잔여를 연구한다.

### Narrative

여러 Episode를 거치며 미래의 주의, 기대, 선택, 복귀점을 조직하는 느린 구조가 무엇인지 탐구한다.

### 자기 피드백

후회·수치·만족·합리화·책임 수용이 다음 행동분포를 실제로 어떻게 바꾸는지 연구한다.

### 재저자화

과거 발생 사실은 보존하면서도 현재의 의미·책임·자기 소유 관계가 다시 판정되는 과정을 연구한다.

### 영향과 권한

느낌과 서사는 행동을 바꿀 수 있지만 외부 사실과 타인의 권리를 자동 확정하지 않는다. 이 경계가 수행에서 어떻게 나타나는지 본다.

### 희소성

시간, 주의, 피로, 인지 통제, 관계 위험이 선택과 고착, 복귀에 어떤 비용 구조를 만드는지 연구한다.

### 타자 재귀

상대를 추정하고, 상대가 자신을 어떻게 볼지 추정하고, 실제 반응을 통해 그 추정을 수정하는 과정을 연구한다.

### 상태와 표현

내부 변화가 관계와 위험 속에서 어떻게 숨겨지고, 새어 나오고, 행동과 말투로 투영되는지 연구한다.

## 6. 연구에서 프롬프트 모델로 내려가는 과정

### 6.1 연구 질문을 하나의 수행 문제로 바꾼다

예:

```text
반복된 작은 실망이 특정 사건 뒤 관계 전체의 해석을 바꾸는가?
```

### 6.2 필요한 작동 원리를 적는다

```text
작은 사건은 즉시 전체 관계를 결정하지 않는다.
반복은 다음 해석에 잔여를 남긴다.
점화 사건은 누적된 잔여와 결합한다.
전환 뒤에도 이전 경로로 복귀하려는 힘이 남는다.
```

### 6.3 LLM이 실제로 수행할 수 있는 지시로 번역한다

연구 개념을 설명하게 만드는 대신, 대화와 행동 선택 속에서 작동하게 만든다.

### 6.4 하나의 프롬프트를 완성한다

완성된 프롬프트는 `models/<model-name>/prompt.md`에 들어가며 하나의 독립 모델이 된다.

### 6.5 평가에서 확인할 열린 질문을 남긴다

정답표를 만드는 대신, 모델을 굴릴 때 관찰할 핵심 긴장을 적는다.

## 7. 모델 평가를 연구로 되돌리는 과정

평가가 끝나면 연구 문서는 다음을 받는다.

### 관찰

실제 응답과 장기 흐름에서 무슨 일이 일어났는가.

### 현상

그 관찰이 어떤 구조적 문제를 드러냈는가.

### 경쟁 원인

연구 가설, 프롬프트 구현, 기반 LLM, 평가 해석 가운데 어떤 원인이 가능한가.

### 다음 질문

다음 모델 또는 다음 즉흥 평가에서 무엇을 더 확인해야 하는가.

### 연구 결정

기존 개념을 유지, 분해, 결합, 수정, 폐기할지 결정한다.

예:

```text
관찰:
모델은 상대의 사과를 받아들였지만 이후에도 모든 모호한 행동을 배신으로 해석했다.

현상:
표면적 관계 회복과 느린 기대 구조가 분리되어 움직였다.

연구 질문:
Narrative의 변화와 현재 관계 표현은 서로 다른 시간척도인가?
```

## 8. 임시 종합이 하는 일

임시 종합은 여러 연구를 한 번에 완성된 이론으로 만들기 위한 문서가 아니다. 다음 세 가지를 위해 쓴다.

1. 현재까지 반복된 구조를 한눈에 본다.
2. 아직 서로 충돌하는 부분을 드러낸다.
3. 다음 프롬프트 모델이 실제로 무엇을 시험해야 하는지 정한다.

좋은 종합은 매끄러운 결론보다 다음을 남긴다.

- 현재 가장 강한 공통 구조
- 서로 다른 해석이 갈리는 지점
- 실제 수행에서 판별 가능한 차이
- 다음 모델에 넣을 최소 구조
- 다음 평가에서 가장 먼저 찌를 약점

---

## 9. 폐기 연구의 역할

`research/discarded/`에는 현재 방향에서 사용하지 않는 연구를 보존한다.

폐기 기록은 다음을 남긴다.

- 무엇을 설명하려 했는가
- 왜 유망하다고 보았는가
- 어떤 모델이나 평가에서 문제가 드러났는가
- 무엇이 이 생각을 대체했는가
- 어떤 부분은 여전히 사용할 수 있는가
- 어떤 조건에서 다시 검토할 수 있는가

이 기록은 실패를 연구 기억으로 바꾸고, 이후 비슷한 아이디어가 다시 나타났을 때 과거 문제를 함께 볼 수 있게 한다.

---

## 10. 연구가 더 단단한 이론으로 자라는 조건

어떤 구조가 다음 과정을 반복해서 견디면 더 높은 이론적 정리의 후보가 된다.

```text
여러 기존 자료에서 반복적으로 발견됨
→ 서로 다른 설명과 비교됨
→ 프롬프트로 실제 구현 가능함
→ 여러 즉흥 평가에서 기능함
→ 제거하거나 바꾸었을 때 수행 차이가 드러남
→ 새 상황에서 어떤 반응이 나올지 더 잘 예상하게 함
```

그때 필요한 이론적 집은 실제로 살아남은 구조의 관계를 따라 만들어진다.

## 11. 연구 문서를 읽고 쓰는 기준

연구 문서를 읽을 때 다음 질문을 사용한다.

- 이 문서는 어떤 현상을 설명하려 하는가?
- 그 현상을 실제 수행에서 어떻게 볼 수 있는가?
- 기존 개념과 무엇이 다른가?
- 어떤 반응이 나오면 이 설명이 약해지는가?
- 어떤 프롬프트 모델과 연결되는가?
- 어떤 평가 기록이 이 질문을 다시 열었는가?
- 다음에 무엇을 시험해야 하는가?
