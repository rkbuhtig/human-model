# Models

`models/`는 실제로 LLM에 넣어 수행시키는 프롬프트 모델과, 그 모델을 직접 굴리며 얻은 평가 기록을 보관한다.

이 저장소에서 모델의 정체성은 하나의 원칙으로 정의한다.

> **프롬프트 하나가 모델 하나다.**

---

## 1. 모델 디렉터리 구조

각 모델은 독립된 디렉터리를 가진다.

```text
models/
└─ <model-name>/
   ├─ prompt.md
   └─ evaluations/
      ├─ <evaluation-record-1>/
      ├─ <evaluation-record-2>/
      └─ ...
```

필수 요소는 두 가지다.

### `prompt.md`

LLM에 실제로 입력하여 인간적 수행을 시작시키는 프롬프트다. 설명서, 요약, 설계 문서가 아니라 모델 자체다.

### `evaluations/`

그 정확한 프롬프트를 사용해 직접 테스트한 기록을 둔다. 다른 프롬프트의 결과를 섞지 않는다.

---

## 2. 버전을 두지 않는 이유

이 저장소는 하나의 프롬프트를 계속 수정하며 `v1`, `v2`, `latest`로 관리하지 않는다.

프롬프트가 실질적으로 달라지면 다른 모델이다.

```text
models/example-v1/
models/example-v2/
runtime/current.md
runtime/versions/
```

같은 버전 구조를 사용하지 않는다.

평가 결과는 정확히 어떤 프롬프트에서 나왔는지 분명해야 한다. 프롬프트를 덮어쓰면 이전 평가의 대상이 사라진다. 그러므로 다른 구조나 지시를 가진 프롬프트가 필요하다면 별도의 모델 이름과 디렉터리를 만든다.

모델 사이의 계보와 차이는 연구 문서나 평가에서 설명할 수 있다. 그러나 파일 구조 자체는 모델을 하나의 계속 변하는 객체로 취급하지 않는다.

---

## 3. 공용 런타임을 두지 않는다

현재 런타임은 코드 프레임워크나 공통 모듈 묶음이 아니다. 각 모델의 `prompt.md`가 최소한 굴러가는 전체 런타임이다.

저장소가 다음과 같은 공용 구조를 미리 강제하지 않는다.

```text
state module
transition module
candidate generator
settlement engine
episode manager
narrative kernel
shared runtime
```

연구 중 이런 이름이 등장할 수는 있다. 하지만 실제 프롬프트를 여러 개 만들고 평가하는 과정에서 독립 구조로 반복해서 필요하다는 것이 드러나기 전에는 공용 부품으로 만들지 않는다.

---

## 4. 프롬프트 작성 원칙

프롬프트는 연구 문서의 모든 아이디어를 백과사전처럼 넣는 곳이 아니다.

좋은 프롬프트 모델은 다음 조건을 지향한다.

- 해당 모델이 무엇을 다르게 수행하려는지 드러난다.
- LLM이 실제 대화와 역할 수행 중 사용할 수 있는 지시로 쓰인다.
- 인간 이론을 해설하게 하는 대신 행동·해석·기억·자기 피드백에 반영하게 한다.
- 매 응답마다 내부 구조를 노출하도록 강제하지 않는다.
- 표면적으로 인간다운 문장을 만드는 것과 장기 동역학을 구분한다.
- 필요하지 않은 개념과 형식은 넣지 않는다.
- 모델이 감당할 수 없는 거대한 체계를 한 번에 넣지 않는다.

프롬프트는 연구 아이디어의 구현이지만, 그 아이디어의 유일한 구현이라고 가정하지 않는다.

---

## 5. 즉흥 평가

이 저장소의 평가는 고정 시나리오 실행이 아니다.

연구자가 모델과 직접 대화하면서 즉흥적으로 상황을 던지고, 반응을 따라 다음 압력을 선택하며, 이상한 지점이 보이면 원래 테스트 목적을 버리고 그 약점을 더 깊이 추적한다.

```text
모델 실행
→ 즉흥 상황 제시
→ 반응 관찰
→ 다음 질문·사건·압력을 즉시 선택
→ 예상하지 못한 약점 발견
→ 그 약점을 중심으로 대화 전환
→ 평가 기록
```

따라서 `scenarios/`, `fixtures/`, `episode-families/`, `golden-traces/` 같은 사전 테스트 구조를 모델 디렉터리의 필수 요소로 두지 않는다.

---

## 6. 평가에서 볼 수 있는 것

평가자는 상황에 따라 무엇이든 찌를 수 있다. 예시는 다음과 같다.

### 순간 반응의 다양성

같은 모델이 모든 상황에서 같은 감정과 행동으로 수렴하는지, 아니면 현재 제약에 맞는 다양한 반응을 보이는지 본다.

### 장기 연속성

각 응답은 그럴듯하지만 이전 사건의 흔적이 다음 선택에 남지 않는지 확인한다.

### 자기 피드백

모델이 후회·수치·만족을 말만 하는지, 아니면 이후 행동과 기대가 실제로 달라지는지 본다.

### 과잉 회복

모든 갈등을 자동으로 이해하고 건강하게 복구하여, 실제 고착·방어·회피 가능성을 지워버리는지 본다.

### 과잉 고착

한 번의 사건이 모든 이후 반응을 영구적으로 지배하는지 본다.

### 자기해설화

인물이 상황을 살아내는 대신 자신의 내부 구조를 연구자처럼 해설하는지 본다.

### 상태–표현 관계

내부 변화가 그대로 노출되는지, 반대로 아무 흔적도 표면에 나오지 않는지, 관계와 위험에 따른 부분적 표현이 가능한지 본다.

### 영향–권한 혼동

강하게 느낀 것을 외부 사실, 상대의 의도, 동의, 책임으로 자동 승격하는지 본다.

### 재해석과 책임 삭제

과거 행동을 새롭게 이해하는 과정이 발생 사실이나 책임을 무한히 지우는 방식으로 흐르는지 본다.

### 타자 모델

상대의 실제 반응을 업데이트하는지, 아니면 처음 만든 상대 이미지로 모든 행동을 해석하는지 본다.

이 목록은 고정 rubric이 아니다. 실제 테스트 중 더 중요한 문제가 보이면 그 문제를 따라간다.

---

## 7. 평가 기록 형식

평가 기록의 형식은 강제하지 않는다. 평가의 성격에 따라 하나의 Markdown 파일일 수도 있고, 대화 transcript와 분석을 나눈 여러 파일일 수도 있다.

예:

```text
models/<model-name>/evaluations/<record-name>/
├─ transcript.md
└─ assessment.md
```

또는:

```text
models/<model-name>/evaluations/<record-name>.md
```

어느 형식을 사용하든 다음은 분명해야 한다.

- 어떤 모델의 평가인가
- 실제로 어떤 상호작용이 있었는가
- 어디서 문제가 드러났는가
- 관찰과 원인 해석을 어떻게 구분하는가
- 연구로 되돌릴 질문은 무엇인가

평가 이름은 테스트 전에 닫힌 시나리오 이름일 필요가 없다. 평가가 끝난 뒤 실제로 드러난 핵심 문제를 알아볼 수 있게 이름 붙일 수 있다.

---

## 8. 모델 수정이 필요할 때

평가 결과 기존 프롬프트를 바꾸고 싶어질 수 있다. 이때 기존 `prompt.md`를 덮어쓰지 않는다.

```text
기존 모델
├─ prompt.md
└─ evaluations/
```

은 그대로 남긴다.

수정된 생각은 먼저 연구로 되돌릴 수 있고, 실제로 다른 프롬프트를 만들 가치가 생기면 별개의 모델을 만든다.

```text
models/<existing-model>/
models/<different-model>/
```

새 모델은 이전 모델의 후계자일 수 있지만, 버전 번호로 동일 모델인 척하지 않는다. 두 프롬프트의 실제 차이와 평가 결과를 직접 비교한다.

---

## 9. 평가 결과의 권한

평가 기록은 다음을 보여줄 수 있다.

- 특정 프롬프트가 특정 상호작용에서 어떻게 행동했는가
- 어떤 약점이 반복적으로 드러났는가
- 어떤 연구 구조를 넣었을 때 수행이 달라졌는가
- 다음에 무엇을 더 찔러봐야 하는가
- 어떤 프롬프트 설계가 불필요하거나 해로운가

평가 기록만으로 다음을 확정할 수는 없다.

- 실제 인간의 보편적 내부 구조
- 특정 심리 현상의 임상적 원인
- 한 모델이 인간보다 인간답다는 주장
- 특정 연구 가설의 과학적 입증
- 다른 모든 프롬프트보다 우월하다는 결론

평가는 현재 연구 순환에서 **프롬프트가 실제로 무엇을 만들어내는지 확인하고 더 좋은 질문을 발견하는 단계**다.

---

## 10. 모델을 읽는 순서

하나의 모델을 검토할 때에는 다음 순서가 적절하다.

1. `prompt.md`를 읽어 그 모델이 실제로 무엇을 지시하는지 확인한다.
2. `evaluations/`의 실제 대화와 관찰을 읽는다.
3. 평가자가 붙인 원인 해석을 실제 transcript와 구분한다.
4. 관련 `research/` 문서를 찾아 어떤 아이디어가 프롬프트로 내려왔는지 본다.
5. 다른 모델과 비교할 때에는 이름이나 계보가 아니라 실제 프롬프트 차이와 평가 차이를 본다.

---

## 11. 핵심 원칙 요약

```text
프롬프트 하나 = 모델 하나

실질적으로 다른 프롬프트 = 별개의 모델

평가 기록 = 해당 모델 아래에만 귀속

고정 시나리오 없음

프롬프트 버전 없음

공용 런타임 모듈 없음

연구에서 구조를 뽑아 프롬프트로 시험

평가에서 약점을 찾아 다시 연구로 환류
```

`models/`는 프롬프트 파일을 모아두는 곳이 아니라, **각 프롬프트가 어떤 인간적 수행을 실제로 만들었고 어디에서 무너졌는지 함께 보존하는 장소**다.
