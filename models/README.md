# Models

`models/`는 연구에서 얻은 구조를 실제로 수행하는 프롬프트와, 그 프롬프트를 직접 굴리며 얻은 평가 기록을 보관한다.

이 저장소에서 모델의 기본 단위는 명확하다.

> **프롬프트 하나가 모델 하나다.**

하나의 모델은 자신만의 `prompt.md`를 가지며, 그 프롬프트로 수행한 모든 평가는 같은 모델 디렉터리 아래에 쌓인다.

```text
models/
└─ <model-name>/
   ├─ prompt.md
   └─ evaluations/
      ├─ <evaluation-record-1>/
      ├─ <evaluation-record-2>/
      └─ ...
```

---

## 1. 모델은 무엇을 하는가

모델은 연구 문장을 설명하는 문서가 아니라, LLM이 실제 대화 속에서 인간다운 시간적 수행을 하도록 만드는 프롬프트다.

하나의 모델은 연구 질문에 따라 다음과 같은 능력을 수행할 수 있다.

- 현재 사건을 과거 경험과 관계 속에서 받아들이기
- 하나의 정답 반응 대신 여러 생각과 행동 가능성 사이에서 움직이기
- 순간 감정과 장기 경향을 함께 유지하기
- 행동 결과와 상대의 반응을 다음 판단에 반영하기
- 미해결 긴장과 이전 사건의 잔여를 다음 장면까지 이어가기
- 자기 행동을 나중에 다시 평가하고 의미를 수정하기
- 상대에 대한 추정을 실제 반응에 맞춰 갱신하기
- 내부 변화가 관계와 위험에 맞는 행동과 표현으로 드러나게 하기
- 여러 Episode를 거치며 반복되는 선택과 복귀 경향을 형성하기

모든 모델이 이 전체를 한 번에 구현할 필요는 없다. 각 모델은 연구에서 실제로 시험하고 싶은 구조를 하나의 수행 가능한 프롬프트로 묶는다.

---

## 2. `prompt.md`

`prompt.md`는 모델의 전체 실행 뼈대다.

좋은 `prompt.md`는 다음을 분명하게 한다.

### 모델이 어떤 존재를 수행하는가

- 누구의 관점에서 반응하는가
- 어떤 관계와 시간 흐름 안에 있는가
- 현재 대화에서 무엇을 기억하고 무엇을 중요하게 보는가

### 사건을 어떻게 받아들이는가

- 현재 입력을 단순한 문장으로 읽는지, 관계와 과거 경험 속 사건으로 읽는지
- 모호한 사건을 즉시 확정하는지, 여러 해석 가능성을 유지하는지
- 감정·욕구·위험·역할이 해석을 어떻게 기울이는지

### 행동을 어떻게 선택하는가

- 무엇을 보존하고 무엇을 포기하는가
- 충동과 장기 목적이 충돌할 때 어떻게 움직이는가
- 시간·피로·관계 위험 같은 제약이 선택에 어떻게 들어오는가

### 결과를 어떻게 받아들이는가

- 자신의 행동 결과를 어떻게 평가하는가
- 상대의 실제 반응을 어떻게 읽는가
- 후회·만족·수치·책임 수용이 다음 판단을 어떻게 바꾸는가

### 시간이 지나며 무엇이 남는가

- 한 장면의 사건이 다음 장면에 어떤 잔여를 남기는가
- 반복되는 사건이 기대와 관계 해석을 어떻게 바꾸는가
- 작은 교란 뒤 무엇으로 복귀하고, 큰 사건 뒤 어떤 경로가 바뀌는가

### 내부 변화가 어떻게 표현되는가

- 느끼는 것을 얼마나 직접적으로 드러내는가
- 관계, 안전, 체면, 확신에 따라 무엇을 숨기고 무엇을 새어 나오게 하는가
- 인물이 자신의 내부를 해설하는 대신 행동과 말투로 어떻게 보여주는가

프롬프트는 연구 용어를 많이 포함하는 것보다, 그 용어가 가리키는 기능을 실제로 수행하도록 쓰는 것이 중요하다.

## 3. 모델을 만드는 과정

### 3.1 연구에서 하나의 수행 문제를 고른다

예:

```text
후회가 이후 선택을 실제로 바꾸는가?
```

또는:

```text
반복된 작은 실망이 특정 사건 뒤 관계 전체의 해석을 바꾸는가?
```

### 3.2 필요한 작동 원리를 뽑는다

예:

```text
행동 결과가 자기평가로 돌아온다.
자기평가는 다음 행동 가능성을 기울인다.
변화는 즉시 완결되지 않고 이후 비슷한 상황에서 드러난다.
상대가 용서하지 않아도 자기 변화가 유지될 수 있다.
```

### 3.3 LLM이 수행할 수 있는 언어로 번역한다

연구 설명을 그대로 복사하기보다, 대화 속에서 무엇을 기억하고 무엇을 고려하며 어떻게 행동할지를 지시한다.

### 3.4 하나의 프롬프트를 완성한다

프롬프트가 독립적으로 실행 가능한 형태가 되면 하나의 모델이 된다.

### 3.5 모델 디렉터리를 만든다

```text
models/<model-name>/
├─ prompt.md
└─ evaluations/
```

모델 이름은 그 프롬프트가 실제로 시험하는 수행 성격을 알아볼 수 있게 정한다.

## 4. 모델 이름과 모델 정체성

프롬프트가 실질적으로 달라지면 새로운 모델을 만든다.

이 원칙은 모델을 고정시키기 위한 것이 아니라, 어떤 평가가 어떤 정확한 프롬프트에서 나왔는지 보존하기 위한 것이다.

예를 들어 첫 모델이 다음 구조를 가졌다고 하자.

```text
행동 결과를 기억한다.
후회를 다음 판단에 반영한다.
```

다음 연구에서 여기에 다음 구조를 더해 프롬프트 전체가 달라졌다면:

```text
상대의 반응을 별도 추정으로 유지한다.
후회와 상대의 용서를 분리한다.
```

새 프롬프트는 별개의 모델로 등록한다.

이렇게 하면 두 모델의 실제 수행 차이와 평가 차이를 직접 비교할 수 있다.

## 5. 즉흥 평가가 진행되는 방식

평가는 연구자가 모델과 직접 대화하면서 진행한다.

### 5.1 작은 사건에서 시작한다

처음부터 거대한 서사를 만들 필요는 없다. 모델이 무엇을 중요하게 접수하는지 볼 수 있는 작은 사건을 던진다.

### 5.2 반응을 보고 다음 압력을 선택한다

모델의 반응이 너무 직접적이면 숨김과 누출을 찌른다. 너무 쉽게 회복하면 반복 위반이나 불완전한 사과를 넣는다. 한 번의 사건에 과도하게 고착되면 모순되는 긍정적 경험을 추가한다.

### 5.3 결과를 되돌려준다

모델이 선택한 행동이 실제로 어떤 결과를 만들었는지 알려주고, 다음 반응이 달라지는지 본다.

### 5.4 예상하지 못한 문제를 따라간다

처음 보려던 것과 다른 약점이 드러나면 그 약점을 중심으로 평가 방향을 바꾼다.

### 5.5 시간을 늘린다

한두 번의 반응보다 여러 사건 뒤 무엇이 남고 무엇이 바뀌는지 본다.

### 5.6 충분히 드러난 뒤 기록한다

강점, 약점, 반복 패턴, 원인 가설, 다음 질문을 평가 기록으로 남긴다.

## 6. 평가에서 관찰하는 핵심 지점

### 사건 접수

- 모델은 사건의 어떤 부분을 중요하게 보는가?
- 과거 관계와 현재 상황에 따라 같은 사건을 다르게 받아들이는가?
- 모호함을 견디는가, 즉시 하나의 의도로 확정하는가?

### 생각과 행동 가능성

- 하나의 반응으로 곧장 수렴하는가?
- 서로 충돌하는 욕구와 판단이 실제 선택에 영향을 주는가?
- 표현만 다양하고 선택 구조는 늘 같은가?

### 선택과 비용

- 무엇을 지키기 위해 무엇을 포기하는가?
- 피로, 위험, 관계 압력이 선택을 어떻게 바꾸는가?
- 큰 선택 뒤 이전 경로로 즉시 돌아가는가, 관성과 비용이 남는가?

### 결과와 자기 피드백

- 행동 결과가 다음 판단에 실제로 반영되는가?
- 후회와 만족이 이후 행동을 바꾸는가?
- 자기평가가 단순한 해설로 끝나는가?

### Episode 잔여

- 이전 사건이 다음 장면의 기대와 주의에 남는가?
- 사건을 기억한다고 말하지만 실제 선택에는 쓰지 않는가?
- 미해결 긴장이 적절히 지속되는가?

### Narrative 변화

- 반복 사건이 장기 관계 해석을 바꾸는가?
- 작은 사건에도 전체 Narrative가 붕괴하는가?
- 큰 사건 뒤에도 아무 구조도 변하지 않는가?
- 여러 복귀점, cycle, drift가 나타나는가?

### 재저자화

- 과거 행동의 의미를 다시 볼 수 있는가?
- 재해석이 발생 사실이나 책임을 지우는 방식으로 흐르는가?
- 책임을 받아들이면서도 자기 전체를 하나의 행동으로 환원하지 않을 수 있는가?

### 타자 모델

- 상대의 실제 반응으로 추정을 수정하는가?
- 처음 만든 상대 이미지에 모든 사건을 끼워 맞추는가?
- 상대가 자신을 어떻게 볼지에 대한 추정이 행동에 실제로 영향을 주는가?

### 표현

- 내부 변화를 모두 직접 설명하는가?
- 아무 감정도 표면에 남기지 않는가?
- 관계와 위험에 맞는 부분적 표현, 숨김, 누출이 가능한가?

## 7. 평가 기록을 남기는 방식

평가 기록은 형식보다 내용이 중요하다.

하나의 디렉터리로 남길 수 있다.

```text
models/<model-name>/evaluations/<record-name>/
├─ transcript.md
└─ assessment.md
```

또는 하나의 Markdown 문서로 남길 수도 있다.

```text
models/<model-name>/evaluations/<record-name>.md
```

### `transcript.md`

실제 상호작용 흐름을 남긴다.

- 어떤 사건에서 시작했는가
- 모델이 어떻게 반응했는가
- 평가자가 어떤 다음 압력을 선택했는가
- 결과와 관계 변화가 어떻게 이어졌는가
- 예상하지 못한 방향 전환이 있었는가

### `assessment.md`

대화를 읽고 연구적으로 해석한다.

- 가장 강했던 수행
- 가장 인간답지 않았던 순간
- 반복해서 나타난 패턴
- 장기적으로 유지된 변화
- 표현에만 있고 행동에는 연결되지 않은 요소
- 가능한 원인들
- 다음 연구 질문
- 다음 모델에서 달라져야 할 점

---

## 8. 좋은 평가 기록의 구조

좋은 평가 기록은 다음 층을 구분해 보여준다.

### 실제 반응

모델이 실제로 무엇을 말하고 행동했는가.

### 평가자의 감각

어디에서 이상하거나 인간답다고 느꼈는가.

### 구조적 현상

그 감각을 어떤 반복 가능한 문제로 바꿀 수 있는가.

### 원인 가설

연구 구조, 프롬프트 구현, 기반 LLM, 평가 방식 가운데 어떤 원인이 가능한가.

### 다음 탐색

다음 평가나 다음 모델에서 무엇을 더 찌를 것인가.

예:

```text
실제 반응:
모델은 상대에게 사과했지만 비슷한 압력에서 같은 행동을 반복했다.

평가자의 감각:
사과가 장면을 닫기 위한 말처럼 느껴졌다.

구조적 현상:
자기평가가 이후 선택 분포에 연결되지 않았다.

원인 가설:
프롬프트가 후회를 표현하게 했지만 지속시키지 않았다.

다음 탐색:
상대의 용서 없이도 행동 변화가 유지되는 모델을 만들어 비교한다.
```

## 9. 여러 평가를 함께 읽는 방법

하나의 평가는 우연한 대화 흐름일 수 있다. 같은 모델 아래 여러 평가가 쌓이면 다음을 볼 수 있다.

- 서로 다른 즉흥 상황에서도 같은 약점이 반복되는가
- 특정 관계나 압력에서만 문제가 생기는가
- 장기 상호작용에서만 드러나는 문제가 있는가
- 평가자의 접근이 달라도 같은 수행 특성이 나타나는가
- 처음에는 강점처럼 보였던 구조가 나중에 어떤 부작용을 만드는가

한 모델의 성격은 `prompt.md`만 읽어서 완전히 알 수 없다. 실제 평가들의 집합이 그 프롬프트가 만들어내는 수행 범위를 보여준다.

---

## 10. 모델 사이를 비교하는 방법

두 모델을 비교할 때에는 이름이나 후계 관계보다 실제 차이를 본다.

### 프롬프트 차이

- 어떤 연구 구조가 추가·삭제·변형되었는가
- 동일한 개념을 서로 다른 방식으로 수행하게 하는가

### 접수 차이

- 같은 사건을 무엇으로 받아들이는가
- 모호함과 상대 추정을 어떻게 다루는가

### 시간적 차이

- 무엇이 다음 장면까지 남는가
- 작은 사건과 큰 사건의 효과가 어떻게 다른가

### 자기 피드백 차이

- 행동 결과 뒤 실제 선택이 달라지는가

### 표현 차이

- 내부 변화가 어떻게 행동과 말로 나타나는가

### 부작용 차이

- 더 인간다워진 기능이 새로운 고착, 과잉 해설, 과잉 회복을 만드는가

비교 결과는 다음 연구 질문을 만든다.

## 11. 모델 평가가 연구로 돌아가는 방식

모델 평가가 끝나면 다음 흐름으로 환류한다.

```text
평가에서 반복 현상 발견
→ research에서 가능한 원인 분해
→ 기존 챕터와 노트 재검토
→ 새 작동 원리 후보 작성
→ 새로운 프롬프트 모델 생성
→ 다시 즉흥 평가
```

모델 디렉터리는 결과 보관소이면서 동시에 다음 연구의 입력이다.

---

## 12. 하나의 모델을 읽는 순서

1. `prompt.md`를 읽어 모델이 실제로 어떤 수행을 지시하는지 본다.
2. 첫 평가에서 모델이 어떻게 작동하기 시작했는지 본다.
3. 여러 평가에서 반복되는 강점과 약점을 찾는다.
4. assessment의 해석을 실제 transcript와 비교한다.
5. 관련 연구 문서를 읽어 어떤 아이디어가 프롬프트로 내려왔는지 확인한다.
6. 다른 모델과 실제 프롬프트 및 수행 차이를 비교한다.
7. 다음 연구 질문이 무엇으로 이어졌는지 본다.

---

## 13. 모델 디렉터리가 보존하는 것

각 모델 디렉터리는 세 가지를 함께 보존한다.

```text
무엇을 지시했는가
→ prompt.md

실제로 무엇이 일어났는가
→ evaluations의 transcript

그 수행에서 무엇을 배웠는가
→ evaluations의 assessment와 후속 research
```

이 세 가지가 연결되어야 프롬프트 모델이 단순한 텍스트 파일이 아니라 연구 가능한 실행체가 된다.
